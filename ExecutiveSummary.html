Contexto

Por que a criptografia atual não é mais suficiente?

Os algoritmos criptográficos que protegem o sistema financeiro global — RSA, ECDSA e Diffie-Hellman — baseiam-se na dificuldade computacional de fatorar grandes números primos. Um computador quântico suficientemente poderoso, utilizando o Algoritmo de Shor, resolve esse problema em tempo polinomial, tornando vulnerável toda a infraestrutura atual.

O NIST publicou em 2024 os primeiros padrões de Criptografia Pós-Quântica (PQC): ML-KEM (CRYSTALS-Kyber) para troca de chaves e ML-DSA (CRYSTALS-Dilithium) para assinaturas digitais. Este projeto avalia a viabilidade prática desses padrões no Banco Inter.

Análise de Risco

Risco estratégico para o Banco Inter

Quatro vetores de ameaça que exigem ação imediata e planejada.

Collect Now, Decrypt Later

Atores mal-intencionados coletam e armazenam hoje dados cifrados — transações PIX, contratos, senhas — para descriptografá-los no futuro com computadores quânticos. Dados coletados hoje já estão em risco.

Risco: Crítico

Infraestrutura PIX

O PIX processa mais de 150 milhões de transações diárias usando TLS com ECDSA. A ausência de PQC representa vulnerabilidade sistêmica ao sistema financeiro nacional, com impacto direto no BACEN.

Risco: Alto

Conformidade Regulatória

SOx, BACEN (Res. 4.658) e os requisitos Nasdaq de ciber-resiliência exigirão progressivamente comprovação de cripto-agilidade. A não-adoção pode resultar em sanções e bloqueio de operações internacionais.

Risco: Médio-Alto

Cadeia de Suprimentos Digital

APIs de terceiros, parceiros Open Finance e integrações B3 utilizam criptografia clássica. Um único elo vulnerável compromete toda a cadeia de segurança do banco.

Risco: Médio
Análise Técnica

Algoritmos avaliados no benchmark

Seis algoritmos comparados em operações funcionalmente equivalentes sob ambientes cloud e local.

Clássico

RSA-2048

Padrão atual para criptografia assimétrica. Segurança baseada na fatoração de inteiros. Chave pública de 256 bytes. Vulnerável ao Algoritmo de Shor.

Seg. Clássica: 112 bits Seg. Quântica: ~0 bits
Clássico

ECDSA-P256

Assinatura digital baseada em curvas elípticas. Mais eficiente que RSA em tamanho de chave (32 bytes), mas igualmente vulnerável a ataques quânticos via Algoritmo de Shor.

Seg. Clássica: 128 bits Seg. Quântica: ~0 bits
NIST PQC

ML-KEM-512

Mecanismo de encapsulamento de chave baseado em redes cristalinas (Kyber). Nível de segurança 1 NIST. Excelente equilíbrio entre velocidade e segurança para chaves de sessão TLS.

Seg. Quântica: ~128 bits Chave: ~800 bytes
NIST PQC

ML-KEM-768

Variante intermediária do Kyber com nível 3 NIST. Recomendado para aplicações que requerem alta confidencialidade de longo prazo — contratos digitais e dados bancários sensíveis.

Seg. Quântica: ~192 bits Chave: ~1.184 bytes
NIST PQC

ML-KEM-1024

Máximo nível de segurança do Kyber (nível 5 NIST). Indicado para ultra-confidencialidade. Overhead de performance superior, porém justificado em operações críticas de compliance.

Seg. Quântica: ~256 bits Chave: ~1.568 bytes
Híbrido

ML-KEM + ECDSA

Modelo dual-stack recomendado para a fase de transição. Opera PQC e clássico em paralelo: se um falhar, o outro garante a segurança. Estratégia aprovada pelo NCSC britânico e ANSSI francês.

Dupla Proteção Overhead: ~15–25%

Arquitetura

Modelo de transição híbrido

O modelo híbrido combina dois algoritmos em série — se um falhar, o outro ainda protege. Para encapsulamento de chaves: X25519 + ML-KEM-768. Para assinaturas: ECDSA P-384 + ML-DSA-65. A chave resultante é o hash dos dois materiais combinados (HKDF).

  • FFI/CGO — Hot Paths

    Implementado no nível do TLS handshake: X25519 gera shared_secret_1, ML-KEM-768 encapsula shared_secret_2, master_key = HKDF(secret_1 ‖ secret_2). Quebrar qualquer um dos dois não compromete a sessão.

  • Microserviço — Negociação de Protocolo

    API interna que recebe as capacidades do cliente e retorna o melhor protocolo disponível (preferência: hybrid-X25519-MLKEM768; fallback: X25519). Cada negociação é logada — visibilidade total da transição.

  • TLS Proxy — Gateway (nginx/Envoy)

    Configuração via cipher suite: ssl_ecdh_curve X25519MLKEM768:X25519. Tenta híbrido primeiro; recua para X25519 se cliente não suporta. Clientes legados continuam sem alteração.

  • Critério de Saída do Híbrido

    Modo PQC puro só é ativado quando 100% da cadeia de confiança estiver migrada e verificada — incluindo CA raiz, certificados de todos os serviços e clientes. Nunca fazer big bang.

Guia Rápido — Quando Usar Cada Algoritmo

Referência técnica para seleção do algoritmo correto em cada contexto do Banco Inter, conforme o documento de estratégia PQC v1.0.

FIPS 203

ML-KEM-768

Encapsulamento de chaves — padrão principal do Banco Inter. Hot paths: Auth, PIX, Key Wrapping, API Gateway TLS, mTLS. Use via FFI/CGO → liboqs. Substitui RSA-OAEP e ECDH.

Seg. Quântica: ~192 bits SLA < 10ms (hot path)
FIPS 203

ML-KEM-1024

KEM de máxima segurança. DC/Cloud key exchange, Legal Hold, Archives regulatórios com retenção de 10 anos. Latência aceita overhead de +30% — margem de segurança justifica.

Seg. Quântica: ~256 bits Overhead: +30% vs KEM-768
FIPS 204

ML-DSA-65

Assinatura digital — algoritmo universal do Banco Inter. PIX, tokens PASETO v4, docs internos, CI/CD, logs SIEM, aprovações de workflow. Substitui ECDSA e RSA-PSS.

Seg. Quântica: ~192 bits Uso universal de assinaturas
FIPS 205

SLH-DSA-128s

Assinatura de longa vida (hash-based). CA raiz, archives regulatórios (>5 anos), logs privilegiados mensais, legal hold. Baseada exclusivamente em SHA-3 — sem lattices ou curvas elípticas.

Vida útil: 20+ anos Evidência regulatória BACEN

Nunca usar ML-KEM para dados diretamente

ML-KEM é um KEM, não um cipher. Sequência correta: gerar DEK aleatória → AES-256-GCM para dados → ML-KEM-768 para encapsular a DEK (Envelope Encryption).

CA raiz: SLH-DSA é obrigatório

A chave raiz tem vida útil de 10–20 anos. SLH-DSA-128s é baseada exclusivamente em SHA-3 — o nível de conservadorismo correto para uma âncora de confiança que precisa durar décadas.

PASETO v4 como formato PQC-native

JWT com RS256/ES256 não suporta ML-DSA nativamente. Use PASETO v4 para tokens em contextos PQC. Mantenha JWT como fallback durante a transição e migre progressivamente.

HTTP/3 é essencial para PQC em rede

Um certificado Kyber + Dilithium pode chegar a 5KB — 20× maior que RSA. No TCP causa fragmentação e retransmissão. QUIC/HTTP/3 com streams independentes elimina esse problema.

Plano de Ação

Roadmap de implementação

5 fases para migração segura e progressiva — critério de aceite SOx claro antes de avançar em cada etapa. Nenhuma fase é big bang.

Fase 0 · Meses 1–3

Fundação

Instalar liboqs 0.10+ em staging com AVX2 habilitado. Configurar liboqs-go com CGO. Criar CA PQC interna paralela (raiz SLH-DSA-128s + intermediária ML-DSA-65). Criar microserviço de auditoria PQC com hash chain ML-DSA-65. Documentar inventário completo de algoritmos clássicos em uso. Aprovação formal do plano pelo Comitê de Segurança e diretoria.

Critério SOx: Inventário de algoritmos financeiros + evidência de hash chain funcionando
Fase 1 · Meses 4–9

Hot Paths — Auth, PIX e API Gateway

Migrar Auth/IAM e PIX para FFI → liboqs (ML-KEM-768 + ML-DSA-65) em modo híbrido. Migrar API Gateway para TLS Proxy com X25519 + ML-KEM-768. Migrar Key Wrapping para FFI com Envelope Encryption. Deploy de Admission Controller no Kubernetes (rejeitar imagens sem assinatura). Binários e imagens Docker assinados com ML-DSA-65. Rollout gradual: 10% → 25% → 50% → 100% com feature flags.

Fase 2 · Meses 10–18

Infra e PKI

Migrar todos os certificados internos para CA PQC (ML-DSA-65). Migrar mTLS do service mesh para híbrido X25519 + ML-KEM-768. Migrar PAM e Key Escrow para microserviços dedicados. Configurar SIEM com hash chain PQC (ML-DSA-65 por evento + SLH-DSA-128s diário). Migrar archives regulatórios para SLH-DSA-128s + ML-KEM-1024. Migrar backups frios e charts Terraform/Helm.

Fase 3 · Meses 19–30

PQC Puro

Remover X25519 dos hot paths (ML-KEM-768 puro em Auth, PIX, Gateway). Migrar Open Banking para ML-KEM-768 puro em coordenação com ICP-Brasil e BACEN. Descomissionar CA RSA antiga após migração completa. Migrar tokens para PASETO v4 com ML-DSA-65 puro (remover JWT ES256). Verificação de integridade de 100% dos backups históricos. Audit final de cobertura PQC para certificação SOx anual.

Manutenção e Cripto-Agilidade Contínua

Monitoramento contínuo de vulnerabilidades em ML-KEM, ML-DSA e SLH-DSA. Rotação anual de chaves ML-DSA em serviços críticos (dual-control obrigatório).
